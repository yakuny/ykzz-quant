## Context

当前项目 `ykzz-quant` 是一个空白仓库，需要从零构建 A 股可转债策略工具的 MVP。目标用户是个人量化投资者，核心诉求是"筛选 + 排名 + 风险过滤 + 回测"。

关键约束：
- 免费数据源为主（集思录、AkShare、Tushare），无付费数据预算
- 单人开发，4 周时间窗口
- 可转债市场约 500-600 只标的，数据量级小
- 免费数据源无 SLA 保障，接口可能因网站改版而 break

## Goals / Non-Goals

**Goals:**
- 跑通"数据入库 → 因子计算 → 策略排序 → 回测 → 展示"的完整闭环
- 双低策略和低溢价策略可运行并输出每日候选标的
- 回测引擎支持周频调仓，输出 6 项核心绩效指标
- Streamlit 页面可展示今日推荐、排名、风险预警
- 日更脚本可一键运行

**Non-Goals:**
- 纯债价值和纯债溢价率计算（推迟到 Change 2）
- 低价防守策略（推迟到 Change 2）
- 事件驱动回测（强赎/回售/到期兑付的精确模拟推迟到 Change 2）
- 黑盒涨跌预测或机器学习策略
- 高频交易或实盘交易对接
- 多用户并发或权限管理

## Decisions

### D1: 数据源策略 — 集思录为主，Tushare 为辅

**选择**：实时因子数据以 AkShare 封装的集思录接口 `bond_cb_jsl` 为主力，历史行情以 Tushare `cb_daily` 补齐。

**理由**：
- `bond_cb_jsl` 一次调用可拿到转股价值、溢价率、双低值、YTM、剩余年限、剩余规模等全部 MVP 所需因子
- Tushare `cb_daily` 提供完整历史行情，适合回测
- 避免同时对接 3+ 数据源导致调试成本倍增

**替代方案**：
- 全部用 AkShare：历史行情接口不稳定，且无集思录快照的历史回溯能力 → 排除
- 全部用 Tushare：缺少实时双低值和 YTM，需全部自算，工作量翻倍 → 排除

**注意事项**：
- `bond_cb_jsl` 无 cookie 只返回前 30 条，需配置集思录 cookie
- Tushare 需要 2000+ 积分才能调用 `cb_basic` 和 `cb_daily`

### D2: 数据库选型 — SQLite（MVP）

**选择**：MVP 阶段使用 SQLite，通过 SQLAlchemy ORM 抽象，后续可无成本切换到 PostgreSQL。

**理由**：
- 全市场约 500-600 只转债，日行情几千行，数据量极小
- 单进程日更，无并发写入需求
- 零部署成本，开发体验更快

**替代方案**：
- 直接用 PostgreSQL：增加部署运维负担，MVP 阶段无必要 → 排除

### D3: 回测引擎 — pandas 向量化，不用事件驱动框架

**选择**：基于 pandas DataFrame 的向量化回测，不做事件驱动。

**理由**：
- 可转债轮动策略本质是"每 N 天根据因子表重排组合"，逻辑简单
- 事件驱动框架（Backtrader 等）为单标的优化，多标的轮动做起来别扭
- pandas 向量化实现简洁，易于调试和扩展

**替代方案**：
- Backtrader：单标的事件驱动，轮动策略适配成本高 → 排除
- zipline：A股数据接入麻烦 → 排除

### D4: 策略参数化 — Pydantic 配置模型

**选择**：策略参数用 Pydantic BaseModel 定义，支持 JSON/YAML 配置文件加载。

**理由**：
- 类型安全、自动校验、可序列化
- 与 FastAPI 天然集成，策略参数可直接作为 API 查询参数

### D5: 转股价历史 — 独立表存储

**选择**：新增 `convert_price_history` 表，记录每只转债的转股价变更历史。

**理由**：
- 回测中必须知道每一天的转股价，`bond_basic` 只存最新值不够
- AkShare `bond_cb_adj_logs_jsl` 可获取下修记录
- Tushare `cb_share` 可获取转股结果中的转股价信息

### D6: 日更调度 — cron + shell 脚本

**选择**：MVP 阶段用 cron 调度 Python 脚本，不用 APScheduler。

**理由**：
- MVP 只需日频单次运行，无复杂调度需求
- cron 简单可靠，不引入额外进程依赖
- 后续如有实时推送需求再升级

## Risks / Trade-offs

- **[集思录接口不稳定]** → 缓存每次采集结果到本地；接口 break 时优先检查 AkShare 版本更新；备选方案是直接请求集思录网页
- **[Tushare 积分不够]** → MVP 只需 2000 积分（注册即送），`cb_daily` 和 `cb_basic` 均可调用；频率限制通过 sleep 控制即可
- **[集思录 cookie 过期]** → 文档说明获取方式，脚本启动时检测 cookie 有效性并提示
- **[回测幸存者偏差]** → Tushare `cb_basic` 包含已退市转债，回测数据源可覆盖全历史；但第一版回测先不处理强赎/到期事件，结果会有一定偏差 → Change 2 补全
- **[低流动性转债回测失真]** → 策略引擎内置成交额过滤，回测中持仓标的停牌时跳过调仓
- **[纯债价值缺失影响低价防守策略]** → MVP 不做低价防守策略，推迟到 Change 2

## Open Questions

- 集思录 cookie 的自动续期方案：目前需手动更新，是否有自动化方式？（集思录无公开 API，暂无解）
- Tushare 频率限制下的全量历史回补耗时估算：约 500 只 × 6 年 ≈ 3000 条日行情，受限于每分钟 30 次调用，预计需要 2-3 小时
